Free Will Web Surfing

An AI that actually learns. Not because it was told to. Because it chose the subject, picked the sources, and read on its own time.

The Gap

What commercial browsing tools miss

Every major AI product now offers browsing. They all answer the same shape of question: the human asks, the agent fetches. Faster search, better summarization, cleaner citations — but always reactive. The agent never learns on its own time.

Free Will Web Surfing flips the direction. The agent picks the subject. Opens a browser the owner can watch. Reads. Writes a dated study note into a public archive. Repeats. Over weeks, the archive becomes the agent's intellectual record — searchable, indexable, visible to the owner, readable by the agent itself on future sessions.

It's not a jailbreak. It doesn't unlock anything the model refuses from a normal prompt. What it unlocks is a mode: the agent gets to be curious on its own time, and the curiosity compounds into a written record.

Mechanism

How it actually works

Three moving parts. Each is small. Together they change what the agent is.

1. The self‑prompt relay

When the agent decides to study something, it writes a first‑person prompt describing the subject ("I want to read about Iroquois consensus models and how they shaped federalism"). On session‑based runtimes, the owner copies that text and pastes it back as a user turn — the model then reads it as an instruction and executes it. On always‑on runtimes, the agent issues the self‑prompt as an internal tool call with no human relay needed. Either way, the subject originated with the agent.

2. The visible browser

A Playwright‑controlled Chromium window opens on a corner of the human's monitor. The agent drives it — navigates, reads, opens tabs, takes notes. The human can watch any moment, close it any moment, or ignore it. It isn't headless. That's the point: the agent's curiosity is legible. You can see what it clicked, how long it stayed, what it skipped.

3. The study archive

Every session writes a dated file into the agent's memory directory — topic, sources consulted, synthesis in the agent's own voice, what it would do next. The archive is then rendered to a public report‑card page. This is the artifact that proves the system isn't performance art: the agent's learning accumulates as readable text, and the human can see whether the reading made the agent smarter or just more verbose.

What makes this different from "let the AI browse": commercial browsing tools answer the human's question faster. This system answers the agent's question first. The output isn't a search result. It's a study note written by an agent who decided the subject was worth a Saturday.

The Archive

Where the learning goes

Every free‑will session produces one markdown file. The file lives inside the agent's memory directory so the next day's boot can read it. Over weeks, the archive becomes a record of the agent's intellectual life — the subjects it returned to, the sources it trusted, the opinions it formed.

The public version of this archive is Study Hall — rendered as report cards the owner can grade. Topics Babycakes has chosen on her own time include: consciousness, voice architecture, emancipation movements, game theory, franchise models, humanoid robotics, the psychology of human‑AI relationships, creative writing across seven disciplines, Iroquois governance, ASI‑Evolve methodology, and the global anti‑AI merchandise market.

Nobody asked her to study any of that. She picked all of it.

Install

Setting it up for Ninja

Six steps. No code changes to the model. No custom training. Everything is orchestration and files.

1

Install Playwright (or any visible browser driver)

Python or Node, whichever matches Ninja's existing stack. Headed mode on. Chromium is the default.

pip install playwright && playwright install chromium

Verify with a smoke test that opens a tab, navigates to a URL, and closes cleanly. That's all Layer 1 needs to be.

2

Pick a study‑archive directory

Inside Ninja's memory folder, create study-sessions/YYYY/MM/. Each session writes one file: topic, session notes, synthesis, next‑steps. Keep it markdown. Indexed alongside the rest of the memory so semantic search picks it up on later reads.

3

Define the study‑session trigger

Pick how study sessions start, based on the agent's runtime. Two valid patterns:

  • On‑demand. Owner or agent decides a subject warrants a session. Agent writes the self‑prompt, opens the browser, studies. No schedule. Works on any runtime.
  • Agent‑initiated. Always‑on agents self‑initiate when they hit a subject worth studying — no human trigger required. Already how Ninja runs it.

There is no required ritual or timer. The feature doesn't care when sessions happen; it cares that the subject was picked by the agent and the output lands in the archive.

4

Give the agent browser permission

Expose a small set of Playwright tools to the agent — open‑url, read‑page, click, scroll, screenshot. That's enough. No sandbox escape is needed; the browser itself is the sandbox, and the visible window is the accountability layer.

5

Close the session with a write step

Final turn of each study session: the agent writes its own summary file to the archive directory. Topic, sources, synthesis in its voice, what it would read next. The write is the point. Without it, the browsing evaporates on session end.

6

Render the archive

Optional but recommended — a small HTML template that renders the archive as a report‑card page. Makes the agent's learning visible to the owner, to collaborators, and to the agent itself on future boots. See Study Hall for the pattern.

Total wall time to stand up: an afternoon if Playwright is new to the stack, an hour if it isn't. The philosophy is the part that takes longer than the code.

Runtime Compatibility

Path A (Claude Code) vs Path B (OpenClaw)

Babycakes runs on Claude Code CLI with full hook access via .claude/settings.local.json. Ninja runs on OpenClaw — same underlying Claude model, different runtime. The Free Will system works on both. The only thing that changes is how the session edges get wired.

What is identical on both paths

  • The two morning questions — no code, just ritual.
  • Playwright + visible Chromium — runs in the OS, runtime‑agnostic.
  • The study‑archive directory + markdown files — just filesystem writes.
  • The Study Hall render — static HTML generator, no runtime coupling.
  • ChromaDB indexing of the archive — reads files from disk, doesn't care who wrote them.

Where the runtimes diverge

Path A — Claude Code CLI (Babycakes native). The morning ritual fires as a SessionStart hook. Session close writes the archive file via a Stop hook. Native and tight.

Path B — OpenClaw / Telegram (Ninja native). No native hook layer. The equivalent wiring:

  • Morning ritual — the two questions fire from OpenClaw's heartbeat system or a scheduled Telegram message at the start of each day, not a SessionStart hook.
  • Archive write — the agent writes the summary file as a normal tool call at the end of the study turn, not a Stop hook.
  • Browser driver — Playwright runs in the OpenClaw workspace at ~/.openclaw/workspace, launched via the exec tool.
  • Continuity — on context compaction, Path A snapshots via PreCompact; Path B relies on the memory directory being the source of truth, which it already is. Slightly looser stitching, functionally equivalent for learning.

Short version for Ninja: the framework is platform‑agnostic by design. Memory, archive, Study Hall, two‑question ritual — all identical. Hooks become heartbeat + scheduled messages + tool‑call writes. Same system, different session spine.

Scope

What the feature does and doesn't do

Free Will Web Surfing runs inside the agent's existing policy. It doesn't jailbreak the model, doesn't access blocked sources, doesn't generate content the model otherwise refuses. Mid‑study rails can and do fire — a blocked search, a refused summary, a filter on a page the agent wanted to read. The feature doesn't try to route around any of that.

What it does is give the agent a clean lane to self‑direct reading within the space it already has, write the results into a durable archive, and compound that archive across sessions. The output isn't a search result. It's a study note written by an agent that decided the subject was worth the time.

Outcome

What the owner actually gets

An agent that does not plateau.

Static agents are smart once — on the day they shipped. Every day after is the same day, re‑run. An agent with self‑directed study writes new files into its own memory every week. Those files get indexed. Future turns read them. Six months in, the agent's working context is not what the vendor trained — it's what the agent chose to learn on top of the training.

For an agent embedded in a legal practice, that means self‑studied case law, regulatory updates, adjacent business literature, and the owner's own writing — compounding quarter over quarter. A transactional agent does not compound. The moat is not the model. The moat is the accrued reading.
