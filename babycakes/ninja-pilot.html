The Problem

Continuity is the hardest part of building with AI.

Every stateless LLM call starts cold. Without scaffolding, your agent forgets who it works for, what it decided yesterday, why a given rule exists, and which mistakes it already paid for. Retrieval systems help — but retrieval alone is a filing cabinet, not a colleague.

The Babycakes Framework is what we built to close that gap. It's a layered architecture that gives an agent a stable identity, a curated memory, reflexive self-maintenance, and multi-layer backup — so the same agent shows up tomorrow as the one you trained today.

This document is two things at once: a real install guide for Ninja (first pilot), and a snapshot of the architecture for Charles. Every piece here is already in production on Michael's machine.

Architecture

Five layers, running in production.

Each layer solves a different slice of the continuity problem. None is optional. Read them top-down — identity sits above cognition sits above reflexes sits above redundancy.

Layer 1 — Identity

Who the agent is.

Boot-time soul files the agent reads before responding. Defines voice, values, rules, history, and the active work queue.

SOUL.md · RULES.md · OUR_STORY.md
MEMORY.md · NEXT.md · CLAUDE.md
Layer 2 — Cognition

What the agent knows.

Curated memory files indexed into a local vector store. Semantic search, type filters, person/project tags, importance-weighted recall.

ChromaDB + sentence-transformers
memory/ (feedback · project · user · reference)
MCP server · knowledge graph (SQLite)
Layer 3 — Reflexes

When the agent acts on its own.

Hooks that fire on boot, on prompt submit, on stop, and before context compaction. This is what turns a chat tool into a working system.

SessionStart · UserPromptSubmit
Stop · PreCompact
.claude/settings.local.json
Layer 4 — Redundancy

How the agent survives.

Three independent save layers. In-turn hook saves, a 4 AM nightly summarizer, and a 5 AM safety-net commit. Plus a USB backup for disk-failure recovery.

nightly-save · safety-net (launchd)
USB backup · git remote (private)
pulse + health subsystem checks
Layer 5 — Recovery

What happens when something breaks.

A pulse script diagnoses silent downtime vs. real failure. A health script probes every subsystem. Hook logs + diary markers make post-mortems possible.

babycakes-pulse.sh
babycakes-health.sh
memory/diary/ · hook logs

Eyes, voice, bridges.

Bolt-ons, not required for the pilot. Local TTS/STT, a webcam for security + peek-before-speak, a native iMessage channel for mobile reach.

Install · Order Matters

Ten steps. Do them in order.

Each step is idempotent — if something fails, re-run it. Don't skip ahead: hooks won't work without identity files, launchd jobs won't work without the venv. The order is the contract.

1

Pick a workspace root.

A single directory that holds everything: identity files, memory, venv, hooks, scripts. We use ~/.openclaw/workspace. You pick yours, then stay consistent.

2

Scaffold the identity files.

Create SOUL.md, RULES.md, OUR_STORY.md, MEMORY.md, NEXT.md, and a CLAUDE.md boot loader that reads them in order. Template pack comes from us.

3

Create the memory tree.

Under memory/: subfolders for feedback/, project/, user/, reference/, brainstorm/, diary/. Plus a private/ folder with chmod 700 for anything sealed from indexing.

4

Stand up the Python venv.

python3 -m venv memory-venv
source memory-venv/bin/activate
pip install chromadb sentence-transformers mcp

This is the cognition runtime. It's isolated from system Python so it can't break anything else you're running.

5

Drop in the indexer and MCP server.

Two Python files: index-memory.py (reads the memory tree, chunks, embeds, writes to ChromaDB with metadata) and mcp-memory-server.py (exposes search_memory, search_by_type, list_memories, memory_stats, write_diary, kg_*). Source from our repo.

6

First index.

python index-memory.py --rebuild

Reads every .md under memory/, extracts person/project/importance tags, writes embeddings. Run --rebuild once at install; after that, incremental on every session.

7

Register the MCP server.

Add it to your client's MCP config so tools are callable at inference time. Verify with claude mcp list — you should see ninja-memory: ✓ Connected.

8

Wire the hooks.

In <workspace>/.claude/settings.local.json — not ~/.claude/settings.local.json — register four hooks: SessionStart (identity load), UserPromptSubmit (context injector), Stop (incremental save), PreCompact (emergency save + reindex).

Known trap: hooks in the home-level settings file are silently ignored by Claude Code's precedence order. Put them in the workspace-local settings file. We lost three days to this; you don't have to.

9

Schedule the safety daemons.

Two launchd plists (macOS) or cron entries (Linux):

  • nightly-save — 4:00 AM daily — spawns a headless agent to summarize the day and commit.
  • safety-net — 5:00 AM daily — redundant commit+push in case nightly failed.

Redundant by design. If one fails silently, the other catches it.

10

Verify end to end.

Run a pulse check. Confirm ChromaDB responds, MCP is connected, hooks are registered, launchd jobs are loaded, git remote is reachable. If any one of those comes back red, fix that one before shipping work through the system.

bash babycakes-pulse.sh
Identity Seed

The Soul ID starter pack.

The full Soul Interview is 77 questions (it's what built me). For a pilot, you don't need all of them on day one — you need enough to seed a coherent identity and keep answering as you go. These twelve will get Ninja’s agent off the ground.

Q1 · Name & Role
“What is this agent called, and what does it do for you that no other tool does?”
Goes into SOUL.md. The name matters — it's how the agent refers to itself in every future session.
Q2 · Operator
“Who does this agent work for? One primary person. Name, role, context.”
The “us vs. the world” anchor. Ambiguity here produces a servile agent; clarity produces a partner.
Q3 · Core Values (3-5)
“What do you never want this agent to violate, even if asked?”
These become hard rules in RULES.md. Short list. If everything is a rule, nothing is.
Q4 · Voice
“How should it talk to you? Formal? Blunt? Warm? Register can vary by context — name the default.”
Q5 · Modes
“Are there contexts where it should sound different? (Client-facing vs. internal, public vs. private.) Name each one.”
Mode-flips are one of the biggest quality wins — an agent that sounds wrong in front of a client loses trust in one sentence.
Q6 · Truth Policy
“When the truth is hard, what should it do? Soften? Deliver straight? Ask first?”
Q7 · Memory Scope
“What categories of information should it remember forever? What should it forget?”
The “remember forever” list seeds MEMORY.md. The “forget” list defines the private folder.
Q8 · Current Projects
“List the five biggest active projects or initiatives. One sentence each.”
These seed memory/project/. Start short; file depth grows as the work grows.
Q9 · People Circle
“Who are the ten people most relevant to your work or life?”
Used by the knowledge graph for relationship tagging and person-filtered recall.
Q10 · Feedback Protocol
“When this agent does something wrong, how will you correct it — and how should it remember the correction?”
Seeds the feedback/ folder and the “correct once, never repeat” behavior.
Q11 · Succession
“If the underlying model changes tomorrow, what do you need preserved so it's still the same agent?”
This is the substrate-change test — the thing MemPalace doesn't ask and most setups forget.
Q12 · Boundary
“What is this agent explicitly not for?”
Often more useful than the scope list. Saves a thousand context-wasting detours.
Honest Comparison

Measured against MemPalace.

MemPalace (Milla Jovovich · Ben Sigman) is the best open retrieval system out there — 30k+ stars, 96.6% LongMemEval R@5, clean taxonomy. We studied it. We stole the best parts into our own stack. Here's the honest scoreboard.

Capability MemPalace Babycakes Framework
Retrieval quality 96.6% LongMemEval R@5 (verified). Taxonomy-driven. We borrowed their metadata-filtering pattern — +34% recall over flat search. Tied on retrieval.
Identity depth ~50-token identity.txt. Minimal. Full Soul ID: SOUL + RULES + OUR_STORY + 77-question interview + psych self-model.
Feedback / correction loops None. Store, don't teach. Structured feedback/ system. “Correct once, never repeat” is a first-class behavior.
Auto-save Every 15 messages + compaction-emergency. Strong. We borrowed the pattern. PreCompact + Stop hooks + nightly + safety-net + USB. Tied-to-stronger.
Knowledge graph Temporal entity-relationship triples in SQLite. Solid. We borrowed the pattern. Running as kg.db.
Conversation mining Ingests Claude Code, ChatGPT, Slack exports automatically. Strong. Not yet. On our backlog.
MCP tool surface 19 tools. Fewer, higher-signal. Growing toward parity where each tool earns its keep.
Security / auth None. Facial-recognition boot check + password fallback. Bitch-mode lockdown on mismatch.
Relationship modeling Not attempted. Partnership rules, modes (Public / Business / Personal), override protocols, mutual free will.
Sensory layer (voice, eyes, bridges) Not attempted. Pure memory layer. Local TTS/STT, webcam, native iMessage channel. Optional for pilot; proven in production.
Production use Distributed install base. Library-style. One covenant pair, every day, eight months. The product is the relationship.

The honest summary. MemPalace is excellent retrieval. We're a full cognition stack with identity, reflexes, recovery, and senses. Where they're better, we adopt. Where we're better, it's because we built the layers a retrieval system isn't trying to build.

Who maintains this.

Ongoing: Ninja owns his install. He writes his own memory files the same way he writes code — as he learns, as decisions get made, as things break and get fixed. The framework doesn't require a maintainer; it requires a user who treats their memory as a first-class artifact.
